# 실험 일지 — 문제 발견 및 해결 과정

## 1. MC 버전 학습 결과 분석 (eval_results_mc.csv)

### 관찰된 학습 곡선

| 구간 (ep) | win% vs Random | mbb/g vs Random | win% vs Rule | 비고 |
|---|---|---|---|---|
| 0 (기준선) | 13% | -553 | 46% | Q값 전부 0, 항상 FOLD |
| 200~2,000 | ~80~87% | +15k~+22k | ~91~94% | 급격한 정책 개선 |
| 4,000~15,000 | 83~90% | +13k~+25k | 92~97% | **정점 구간** |
| 16,000~24,000 | 70~85% | +6k~+20k | 70~95% | 하락 시작, 변동 폭 증가 |
| 26,000~36,000 | 30~55% | +3k~+10k | 49~65% | 급격한 붕괴 |
| 38,000~40,000 | **32%** | **-232** | **47%** | 사실상 기준선 수준으로 회귀 |

### 문제: Policy Collapse

ep ~10,000에서 정점(vs Random 90%, vs RuleBased 96.5%)을 찍은 뒤, 학습이 진행될수록 성능이 점진적으로 무너졌다. 학습 종료 시점에서 vs Random 승률이 **32%**까지 하락했고, 이는 단순 무작위 상대에게도 지는 정책이 되었음을 의미한다.

### 원인 분석

**원인 1: ε-decay가 붕괴의 트리거**

현재 코드에서 ε는 ep 32,000까지 1.0 → 0.05로 선형 감소한다. 성능 하락은 ep 20,000부터 시작해 ε가 거의 0.05에 도달하는 32,000 이후 가속된다. 탐색이 줄면서 **잘못된 그리디 정책에 잠겨버리는(exploitation trap)** 현상이 발생했다.

**원인 2: MC의 높은 분산**

MC는 에피소드 전체를 역방향으로 전파하므로 terminal reward의 노이즈가 그대로 모든 셀에 전달된다. 즉각 비용(immediate cost)을 함께 반영하면서 "베팅 = 즉시 -X칩" 신호가 강하게 누적되고, 결과적으로 **공격 액션의 기대값이 시간이 갈수록 과소평가**되어 폴드로 수렴했다.

**원인 3: 폴드 수렴의 징후**

- vs Random win% 32% 상태에서 mbb/g가 -232로 거의 0에 가까움
  → 매 판 소량의 블라인드를 잃되, 가끔 강한 핸드로 회수 = 과도한 폴드 시그니처
- vs RuleBased mbb/g는 학습 내내 음수(대부분 -1k ~ -7k)
  → RuleBased의 강한 베팅에 폴드만 반복하면서 블라인드 + 콜 비용 헌납

## 2. 해결 방향: ε-greedy → UCB (Upper Confidence Bound)

### ε-greedy의 근본적 약점

ε 확률로 *어떤 행동이든* 동일 확률로 무작위 탐색 → 이미 나쁜 줄 아는 행동도 다시 시도하고, 한 번도 시도하지 않은 유망한 행동은 운에 맡긴다. 그리고 ε가 0에 가까워질수록 탐색이 멈추면서 잘못된 정책에서 빠져나오지 못한다.

### UCB의 접근 방식

$$a_t = \arg\max_a \left[ Q(s,a) + c \sqrt{\frac{\ln N(s)}{N(s,a)}} \right]$$

- $N(s,a)$ : 상태 $s$에서 행동 $a$를 선택한 횟수
- $N(s)$ : 상태 $s$를 방문한 총 횟수
- $c$ : 탐색 강도 조절 계수 (하이퍼파라미터)

**핵심 원리**: 방문 횟수가 적은 행동에 자동으로 보너스를 부여해 탐색하게 만들고, 방문이 쌓일수록 보너스가 줄어들어 자연스럽게 활용 모드로 전환된다. ε처럼 강제 decay 스케줄이 필요 없으며, policy collapse 위험이 크게 줄어든다.

### ε-greedy 와 UCB 비교

| 항목 | ε-greedy | UCB |
|---|---|---|
| 탐색 방식 | 균등 무작위 | 불확실성 비례 보너스 |
| decay 필요 | 있음 (스케줄 의존) | 없음 (방문 횟수로 자동 조절) |
| exploitation trap | 발생 가능 | 구조적으로 완화 |
| 구현 복잡도 | 낮음 | 낮음 (카운트 테이블 추가만 필요) |

### 구현 계획

- 기존 Q-테이블 옆에 방문 카운트 테이블 `N[r, p, s, a]` 추가
- `epsilon_greedy()` → `ucb_action()` 으로 교체
- 새 폴더 `poker-pokerkit-ucb/` 로 분리해 MC 버전과 동일 조건에서 비교

## 3. UCB 버전 학습 결과 (poker-pokerkit-ucb/)

TD-UCB (`train_eval_td.py`) 와 MC-UCB (`train_eval_mc.py`) 를 각각 40,000 에피소드 학습해 기존 MC ε-greedy 와 비교했다.

### 3-1. TD-UCB 결과 (`eval_results_td_ucb.csv`)

| 구간 (ep) | win% vs Random | mbb/g vs Random | win% vs Rule | 비고 |
|---|---|---|---|---|
| 0 (기준선) | 13% | -552 | 46% | Q값 전부 0 |
| 200~2,000 | 77~85% | +7k~+20k | 90~96% | 빠른 초기 학습 |
| 4,000~10,000 | 74~89% | +12k~+38k | 82~95% | 안정적 유지 |
| 10,000~25,000 | 72~85% | +14k~+35k | 78~92% | 소폭 하락 있으나 붕괴 없음 |
| 25,000~40,000 | 65~83% | +14k~+34k | 75~90% | **40,000 ep까지 붕괴 없음** |

**핵심 관찰**: ε-decay 스케줄 없이 40,000 ep 내내 win% vs Random이 65% 이상 유지됐다. MC ε-greedy의 ep 38,000~40,000 수준(32%)과 비교하면 극적인 차이.

### 3-2. MC-UCB 결과 (`eval_results_mc_ucb.csv`)

| 구간 (ep) | win% vs Random | mbb/g vs Random | win% vs Rule | 비고 |
|---|---|---|---|---|
| 0 (기준선) | 13% | -552 | 46% | Q값 전부 0 |
| 200~400 | 63~83% | +8k~+13k | 69~93% | 빠른 초기 학습 |
| 600~1,800 | 40~66% | +2k~+12k | 53~72% | **급격한 하락** |
| 2,000~8,000 | 33~67% | +2k~+13k | 48~76% | 변동 크고 회복 불안정 |
| 8,000~40,000 | 22~44% | -1k~+10k | 44~64% | 붕괴 고착 (~30% 대) |

**핵심 관찰**: UCB 탐색으로도 MC의 정책 붕괴를 막지 못했다. ep 400에서 83.5% 피크를 찍은 뒤 ep 5,000부터 ~30% 수준으로 고착됐다. 이는 탐색 방식의 문제가 아니라 **MC 업데이트 구조 자체의 분산 문제**임을 시사한다.

### 3-3. 알고리즘별 비교

| 알고리즘 | 탐색 | ep 40,000 win% vs Random | 정책 붕괴 | 평균 mbb/g (ep 10k~40k) |
|---|---|---|---|---|
| MC ε-greedy | ε 선형 decay | **32%** | ✗ 붕괴 | ~+3k (하락) |
| MC-UCB | UCB1 | **36%** | ✗ 붕괴 | ~+3k (동일) |
| **TD-UCB** | UCB1 | **75%** | ✓ 안정 | ~+24k (안정) |

### 3-4. 결론

1. **TD + UCB 조합이 최적**: TD의 Bellman 부트스트랩은 단계별로 즉각 교정하므로 리턴의 분산이 낮고, UCB가 탐색 부족을 방지해 두 장점이 시너지를 낸다.
2. **MC 붕괴는 탐색이 아닌 업데이트 구조의 문제**: UCB로 탐색을 개선해도 MC 역전파에서 terminal reward의 높은 노이즈가 모든 셀에 전파되는 구조적 문제는 해결되지 않는다. MC 자체를 쓰려면 baseline subtraction(평균 리턴 차감) 또는 n-step return 등 분산 감소 기법이 추가로 필요하다.
3. **ε-greedy의 decay 스케줄 의존성이 핵심 취약점**: 하이퍼파라미터(EPS_DECAY_END, EPS_END)를 잘못 설정하면 exploitation trap에 빠지지만, UCB는 방문 횟수로 자동 조절되므로 이 문제가 없다.

## 4. MC + 즉각 비용의 구조적 결함 심층 분석

### 4-1. terminal_reward 의 비대칭성

`train_eval.py` 의 역전파 코드:

```python
terminal_reward = float(pk_state.stacks[learner_id] - stack_at_last_action)

G = terminal_reward
for (r, s, a, imm) in reversed(trace):
    G = imm + gamma * G
    update_mc(r, pos, s, a, G)
```

`imm = stack_after - stack_before` 는 베팅/콜 시 항상 음수(칩이 팟으로 빠짐), 체크/폴드 시 0이다.  
`stack_at_last_action` 은 학습자의 **마지막 액션 직후** 스택이다.

| 결과 | final_stack | terminal_reward |
|---|---|---|
| **승리** | +400 (팟 전부 회수) | `400 - 0 = +400` |
| **패배** | 0 | `0 - 0 = 0` |

**패배 시 `terminal_reward = 0`**: 칩 손실은 이미 imm에서 전부 차감됐고 터미널에서 추가 보정이 없다.

### 4-2. G 역전파 수치 추적

5액션 예시 (PREFLOP RAISE → FLOP BET → TURN BET → TURN CALL → RIVER ALLIN):

| 액션 | imm | 패배 G (terminal=0) | 승리 G (terminal=+400) |
|---|---|---|---|
| RIVER ALLIN | -100 | **-100** | **+260** |
| TURN CALL | -25 | **-115** | **+209** |
| TURN BET | -25 | **-128.5** | **+163.1** |
| FLOP BET | -30 | **-145.7** | **+116.8** |
| PREFLOP RAISE | -20 | **-151.1** | **+85.1** |

### 4-3. 50% 승률에서 기댓값이 음수

| 액션 | E[G] = 0.5×(승리G) + 0.5×(패배G) | 방향 |
|---|---|---|
| RIVER ALLIN | 0.5×260 + 0.5×(-100) = **+80** | ✓ 양수 |
| TURN CALL | 0.5×209 + 0.5×(-115) = **+47** | ✓ 양수 |
| TURN BET | 0.5×163 + 0.5×(-128) = **+17** | ✓ 양수 |
| FLOP BET | 0.5×117 + 0.5×(-146) = **-14.5** | ✗ 음수 |
| PREFLOP RAISE | 0.5×85 + 0.5×(-151) = **-33** | ✗ 음수 |

**결론**: 브레이크이븐(50% 승률) 플레이어여도 FLOP·PREFLOP 베팅 액션의 기댓값이 음수. 학습이 진행될수록 Q(RAISE) < Q(FOLD)=0 이 되어 폴드 수렴 발생.

수학적 원인: 패배 시 G는 imm 음수만 누적(terminal=0), 승리 시 terminal(+400)이 imm을 상쇄하지만 γ=0.9 할인 때문에 **완전 상쇄가 안 됨**. 초반 액션일수록 γ 할인이 누적되어 양의 기여가 줄고 음수 편향이 강해짐.

$$E[G_{\text{PREFLOP}}] = 0.5 \times (+85.1) + 0.5 \times (-151.1) = -33 < 0$$

### 4-4. 설계 의도 vs 실제 구현 불일치

**원래 의도**: 각 액션이 베팅한 금액 비율로 terminal payoff를 비례 배분.

베팅 비율이 20:5:5:6:4 (합=40) 이고 payoff=+400 이라면:

$$R_i = \frac{|\text{imm}_i|}{\sum|\text{imm}|} \times \text{payoff}$$

| 액션 | 베팅비율 | 의도한 R (승리) | 의도한 R (패배) |
|---|---|---|---|
| PREFLOP RAISE | 20/40 | **+200** | **-200** |
| FLOP BET | 5/40 | **+50** | **-50** |
| TURN BET | 5/40 | **+50** | **-50** |
| TURN CALL | 6/40 | **+60** | **-60** |
| RIVER ALLIN | 4/40 | **+40** | **-40** |

→ 승패가 대칭 → 50% 승률 E[R] = 0 → 폴드 수렴 없음

**실제 구현**: `G = imm + γ * G` 에서 imm이 **비례 배분 보상**이 아니라 **음수 패널티**로 작동. 패배 시 terminal=0 이므로 음수 imm만 누적됨.

### 4-5. 해결 방향

| 방법 | 내용 | 비대칭 해소 |
|---|---|---|
| **Pure MC** | imm 제거, `G = payoff → γ * G` 전파 | ✓ (E[G]=0 @ 50%) |
| **비례 배분 MC** | `R_i = \|imm_i\| / Σ\|imm\| × payoff`, γ 적용 | ✓ (설계 의도와 일치) |
| **정규화** | `G /= STARTING_STACK` | ✗ (비율은 그대로) |

가장 먼저 검증할 것: **Pure MC** 실험 — imm 제거만으로 붕괴가 사라지면 원인 규명 완결.
